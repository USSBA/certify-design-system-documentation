Questionnaire pages

A questionnaire page will have the following items:

  • Breadcrumbs
  • Page title
  • Lead text
  • Questions, see below (one or multiple)
  • Actions
    • Primary (ex: Save and continue)
    • Secondary (ex: Cancel, Go back)
    • Tertiary (ex: Skip)

Questions

A question is a set of components wrapped in a sba-c-question-container.

  • Question
    • Badges

      Use the badges to add flags to questions. NOTE: This concept is still under consideration. This is a placeholder for now.

    • Question primary text

      The question is the primary ask of the user. There must be a direct relationship between the question and the response solicited from the user. Questions should be focused and direct, and should not include text such as "for example."

    • Question support text

      This provides additional context about the nature of a question. Support text can include sample responses, in-depth descriptions, supplemental instructions, definitions, lists, etc., or any important information that the user needs know before providing a response. Quesiton support text will include HTML support for:

      • <p>
      • <ul>
      • <ol>
      • Links, bold text, italicized text
  • Response

    The response is where the user provides a response to the question being asked. The response options must relate directly to the question being asked. The response will support a broad range of inputs, including:

    • Radio buttons
    • Check boxes
    • Text input
      • Text
      • Text area
      • Date
      • Date Range
      • Certify components:
        • Document prompt
        • Table
        • Inputs with ornaments
  • Follow-ups

    Follow-ups remain hidden and disabled on the page and only appear when the user makes a specific response that triggers it to appear. The follow-up may be text or another question or set of questions. Question follow-ups use the same model as primary questions.

  • Question footnotes

    Think of these as a footer, or "details" section for a question. It may include links to the CFR and other information that might be useful to the user, but not the type of information that the user needs to know before determining a response.

Accessibility

It is important to create a meaning relationships between each component of the questionnaire. This can be done in the following ways.

  • If there is a footnote, reference its id with aria-details on the sba-c-question so that the primary text of the question and the supporting text reference the details.
  • For each input in a sba-c-response, use aria-details to reference the id of itâ€™s sba-c-question.
  • Every fieldset must include a legend. If the legend does not need to appear visually, apply the class name of sba-u-visibility--screen-reader to hide it, but make available to screen readers.
  • If a question has a follow-up that is triggered by a response, for example, selecting a specific radio button, include a reference to the follow-up's id with aria-controls and use aria-expanded to indicate whether or not the follow-up is visible.
